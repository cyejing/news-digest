# Agents

维护 `news-hotspots` 时，优先遵循这里的规则。  
面向主会话和执行代理的入口与交付规则，以 [SKILL.md](/Users/chenyejing/project/news-hotspots/SKILL.md) 和 [references/hotspot-prompt.md](/Users/chenyejing/project/news-hotspots/references/hotspot-prompt.md) 为准；本文件只负责项目级维护约束、脚本边界和同步要求。

## 项目定位

- [run-pipeline.py](/Users/chenyejing/project/news-hotspots/scripts/run-pipeline.py) 是统一编排入口。
- [merge-sources.py](/Users/chenyejing/project/news-hotspots/scripts/merge-sources.py) 负责输入归一、评分、去重、topic 分组和 `merged.json` 组装。
- [merge-hotspots.py](/Users/chenyejing/project/news-hotspots/scripts/merge-hotspots.py) 负责生成最终热点 JSON / Markdown，并按 `mode` 归档。
- [source-health.py](/Users/chenyejing/project/news-hotspots/scripts/source-health.py) 只负责读取 `meta/*.meta.json` 做诊断，不回读抓取结果 JSON。
- 维护时统一使用 `<SKILL_DIR>` 与 `<WORKSPACE>` 表达路径语义，不依赖当前 shell 的相对路径。
- 同一台机器上不要并发运行多个热点任务；固定 debug 和归档结构会互相覆盖。

## 当前实现基线

- GitHub 相关环境变量只保留 `GITHUB_TOKEN`；不要重新引入 `GH_APP_ID`、`GH_APP_INSTALL_ID`、`GH_APP_KEY_FILE`。
- `run-pipeline.py` 当前主参数使用 `--archive`、`--mode`、`--hours`；不要把旧的 `--archive-dir`、`--debug-dir`、`--output` 写回主契约。
- `source-health.py` 当前使用 `--input`；不要再写 `--input-dir`。
- 最终归档文件由 `merge-hotspots.py` 按 `mode` 命名，例如 `daily.json`、`daily.md`、`weekly.json`、`weekly1.json`、`weekly.md`。
- 面向用户的最终交付规则只定义在 [references/hotspot-prompt.md](/Users/chenyejing/project/news-hotspots/references/hotspot-prompt.md)；不要在这里重复展开。
- `<LANGUAGE>` 是硬约束：最终交付必须先把归档 Markdown 中的正文完整翻译成目标语言，再按原 Markdown 结构输出；不能因为“原文归档”而跳过翻译。

## 工作区与归档

工作区配置覆盖顺序：

1. `<WORKSPACE>/config/news-hotspots-sources.json`
2. `<WORKSPACE>/config/news-hotspots-topics.json`
3. `<WORKSPACE>/config/news-hotspots-topic-rules.json`
4. `<WORKSPACE>/config/news-hotspots-api-sources.json`
5. `<SKILL_DIR>/config/defaults/`

归档目录：

- `<WORKSPACE>/archive/news-hotspots/<DATE>/json/`
- `<WORKSPACE>/archive/news-hotspots/<DATE>/markdown/`
- `<WORKSPACE>/archive/news-hotspots/<DATE>/meta/`

推荐主命令：

```bash
uv run <SKILL_DIR>/scripts/run-pipeline.py \
  --defaults <SKILL_DIR>/config/defaults \
  --config <WORKSPACE>/config \
  --archive <WORKSPACE>/archive/news-hotspots \
  --mode daily \
  --hours 48 \
  --verbose --force
```

## 脚本职责边界

### `run-pipeline.py`

- 保持“编排层 + 诊断层 + 归档层”分离。
- `main()` 只保留高层流程，不要把 step meta 组装、payload 解析、最终产物复制塞回主函数。
- 最终热点 JSON 和 Markdown 的命名与归档交给 `merge-hotspots.py`。
- `pipeline.meta.json` 继续记录：
  - `hotspots_output`
  - `markdown_output`
  - `cooldown_s`
  - 归档根目录与归档结果
- 外部超时或中断时，应尽量基于已完成步骤继续尝试 merge / hotspots，并写出可诊断的 `pipeline.meta.json`。

### `merge-sources.py`

- 保持“输入归一 → 评分 / 相似性 / 去重 → topic 分组 → 输出组装”的阶段化结构。
- 优先在单文件内部收口，不急着拆新模块。
- 只要已有部分 fetch 结果文件，就应允许继续生成 `merged.json`，不要强依赖所有 fetch 全量成功。

### `merge-hotspots.py`

- 它是最终热点产物生成器，不只是 JSON 渲染器。
- 负责从 `merged.json` 生成最终热点 JSON 和 Markdown。
- 负责按 `mode` 统一命名并归档到 `<DATE>/json/` 与 `<DATE>/markdown/`。
- `ARCHIVED_JSON=` 与 `ARCHIVED_MARKDOWN=` 是上游读取最终产物路径的稳定契约。
- debug 输出只用于调试，不要把 debug JSON 当成最终用户交付物。

### `source-health.py`

- 保持“三层结构”：
  - meta 发现
  - 诊断归一
  - 文本渲染
- 报告结构固定为：
  - `History report`
  - `Run details`
- 如果 `--input` 指向 archive 根目录，应自动聚合最近 7 天 `<DATE>/meta/` 下的元数据。

### `test-news-hotspots.sh`

- 保持统一调度入口，不要重新扩散成多个测试脚本。
- step 脚本映射、输出路径映射、公共参数拼装应集中维护。
- 如果脚本参数变化，优先同步这里，再同步 README / SKILL / references。

### fetch 脚本

- 优先保持 CLI 一致：
  - `--defaults`
  - `--config`
  - `--hours`
  - `--verbose`
  - `--force`
- 同一网站、同一域名默认串行抓取。
- 默认优先靠 cooldown 主动降频，避免触发限流。
- 新抓取脚本应提供可覆盖的 cooldown 环境变量。
- 除非确认目标站点稳定支持更高频率，否则不要加并发。

## 数据与诊断契约

- step 诊断统一优先使用：
  - `items`
  - `call_stats`
  - `failed_items`
- 不要把 `count` 恢复成 pipeline / meta / 日志的主语义。
- 如果 step meta 已经有完整失败诊断，不要再新增第二套错误表达。
- fetch 顶层结果应尽量统一包含：
  - `source_type`
  - `calls_total`
  - `calls_ok`
  - `calls_kind`
  - `items_total`
- 单个 source / topic / repo 结果应尽量统一包含：
  - `status`
  - `items`
  - `count`
- 旧字段可以保留做兼容，但新增逻辑优先消费统一语义字段：
  - `items_total`
  - `calls_total`
  - `calls_ok`
  - `failed_items`

## Topic 与排序规则

默认 topic：

- `ai-frontier`
- `ai-infra`
- `github`
- `technology`
- `business`
- `world`
- `science`
- `social`

topic 正式定义以 `<SKILL_DIR>/config/defaults/topics.json` 为准；新增或调整 topic 时，优先更新对应 topic 的：

- `description`
- `classification.definition`
- `classification.when_to_use`
- `classification.when_not_to_use`
- `classification.examples`

维护时重点遵守：

- source 配置统一使用单值 `topic`，不再使用 `topics` 数组。
- topic 规则优先从 `topic-rules.json` / `news-hotspots-topic-rules.json` 读取，不把分类词表长期散落在脚本里。
- API source 清单优先从 `api-sources.json` / `news-hotspots-api-sources.json` 读取，不把 source + topic 绑定长期硬编码在 `fetch-api.py`。
- 分类优先级固定为：`github` > `ai-infra` > `ai-frontier` > `technology` > `business` > `world` > `science` > `social`。
- 分类原则是“先细分类，后大分类”：
  - 能准确命中 `ai-frontier`、`ai-infra`、`github` 时，不再回落到一级大分类。
  - 只有细分类不适用时，才归入 `technology`、`business`、`world`、`science`、`social`。
- `github` 只表示 GitHub 原生动态，不泛化为全部开源内容。
- `developer-tools` 与 `cybersecurity` 不再独立成 topic，默认归入 `technology`。
- `markets-business` 与 `macro-policy` 不再独立成 topic，统一归入 `business`。
- `world-affairs` 不再独立成 topic，统一归入 `world`。
- `ai-models` 与 `ai-agents` 不再独立成 topic，统一归入 `ai-frontier`。
- `ai-ecosystem` 不再独立成 topic，统一归入 `ai-infra`。
- 每个 source / item 只保留 1 个主 topic；运行时如果命中多个候选 topic，按全局优先级收敛成单个 `topic`。

新增 source 时的 topic 判断流程：

1. 先看 source 的长期主轴，而不是偶发一两篇内容。
2. 先判断能否稳定命中细分类：
   - 持续追踪模型、agents、AI apps、AI lab 发布，归 `ai-frontier`
   - 持续追踪芯片、算力、电力、机器人、智能驾驶、Tesla、SpaceX，归 `ai-infra`
   - GitHub repo、release、Trending、GitHub Blog，归 `github`
3. 如果细分类不适用，再回落一级类：
   - 开发工具、工程实践、网络安全、通用科技媒体，归 `technology`
   - 财经媒体、公司经营、资本市场、产业竞争、宏观政策对企业/市场影响，归 `business`
   - 国际新闻、战争、外交、地缘政治，归 `world`
   - 论文、实验室成果、研究博客、研究驱动型工程突破，归 `science`
   - 教育、就业、未来工作、媒体传播、社会文化影响，归 `social`
4. 如果 source 长期横跨多个方向，只选“最常见、最稳定、最主要”的那个 topic，不做多 topic 配置。
5. 如果 source 只是个别文章碰到某个话题，但长期主轴不在这里，不要因为少量样本改 topic。
6. 判断边界不清时，先回看 `<SKILL_DIR>/config/defaults/topics.json` 里的 `classification` 定义，再决定归类。

评分与去重规则：

- `source priority` 只是轻量基础信号。
- fetch 内互动 / 热度只影响该 fetch 内局部排序。
- 跨 `source_type` 热点有有限加分。
- 历史相似内容会明显降权。
- topic 输出阶段会做来源多样性重排。

RSS 默认池规则：

- 机构媒体、官方博客、公共机构和行业媒体优先默认启用。
- 明显个人博客保留在 `sources.rss` 尾部作为候选源。
- 候选源统一用 `"enabled": false`。
